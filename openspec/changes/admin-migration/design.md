# Design: Admin Migration to Next.js

## Technical Approach
We will build a new Next.js application in `admin-ui/` that serves as the administrative interface. This application will communicate with the existing Go backend via **ConnectRPC**. The Go backend will expose a new `AdminService` and implement a middleware to enforce **RBAC** (Role-Based Access Control). To ensure **visual parity**, we will port existing Tailwind-based styles and HTMX components into React components.

## Architecture Decisions

### Decision: ConnectRPC for Frontend-Backend Communication
**Choice**: Use Connect-ES on the frontend and Connect-Go on the backend.
**Alternatives considered**: REST/JSON, GraphQL.
**Rationale**: ConnectRPC provides type safety, autogenerated clients, and high performance while being compatible with standard HTTP/1.1 and HTTP/2. It aligns with the existing mobile sync infrastructure.

### Decision: JWT with HttpOnly Cookies
**Choice**: Authenticate via JWT, stored in `httpOnly` cookies for the Next.js app.
**Alternatives considered**: LocalStorage, Server-side sessions (Redis).
**Rationale**: Cookies are more secure against XSS. Next.js can read the cookie during Server Side Rendering (SSR) to handle redirects and initial data fetching without flickering.

### Decision: Sector-Based Permissions
**Choice**: A join table `user_sectors` linking users to specific sector IDs.
**Alternatives considered**: Adding a `sector_id` column to the `users` table.
**Rationale**: Multiple sectors can be assigned to a single reader or supervisor, requiring a many-to-many relationship.

## Data Flow

    User (Browser) ──→ Next.js (Admin UI) ──→ ConnectRPC Handler (Go) ──→ Repository (Postgres)
          │                │                      │                       │
          └──── Cookie ────┘                      └──── JWT Validation ───┘

## Interfaces / Contracts

### Proto Definition (Draft)
```proto
service AdminService {
  rpc Login(LoginRequest) returns (LoginResponse);
  rpc GetUsers(GetUsersRequest) returns (GetUsersResponse);
  rpc UpsertUser(UpsertUserRequest) returns (UpsertUserResponse);
  rpc GetSectors(GetSectorsRequest) returns (GetSectorsResponse);
}
```

### DB Schema (Draft)
```sql
CREATE TABLE users (
    id UUID PRIMARY KEY,
    email TEXT UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,
    role TEXT NOT NULL -- 'ADMIN', 'SUPERVISOR', 'READER'
);

CREATE TABLE user_sectors (
    user_id UUID REFERENCES users(id),
    sector_id TEXT NOT NULL,
    PRIMARY KEY (user_id, sector_id)
);
```

## Migration / Rollout
1. Deploy Next.js app in a separate domain/path (e.g., `/v2/admin`).
2. Run SQL migrations for users and roles.
3. Seed the initial ADMIN user.
4. Phase out the legacy `/admin` once visual parity is verified.
